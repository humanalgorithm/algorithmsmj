Django

Django is a web framework that allows developers to write code in Python on the backend. Similar to other web frameworks (such as Ruby on Rails). Django uses a model/view/controller architecture however with a slight difference - Django is model/view/template. Django is centered around using templates in order to display your web pages

In a Django implementation there are a few core files that the developer will modify to bulid the application. These files are:

  • settings.py
  • urls.py
  • models.py
  • views.py

The settings.py file contains configurations for middleware, installed application, directories for templates and database engines to use. For each application you write you will need to specify the settings that match what you are trying to achieve or the application might not run correctly.

The urls.py file uses regular expressions to determine where a request should be routed. Every time a user sends a request to Django it will look at the urls.py and see if there are any matching patterns to route the request. Here is a sample of the urls.py from sorting for this application: 


        from . import views

        urlpatterns = [url(r'^$', views.home, name="home"),
                       url(r'^learnmore/$', views.learnmore, name="learnmore"),
                       url(r'^about-us/$', views.aboutus, name="aboutus"),
                       url(r'^pythonabout/$', views.pythonabout, name="pythonabout"),
                       url(r'^.*/$', views.fourzerofour, name="fourzerofour"),
                       ]

The models.py is where you store information about your models(data). Here you can define the structure of your data for Django to work with. Here is an exceprt from the models.py for this website that defines the signups from the signup form on the main page: 


        class SignUp(models.Model):
            first_name = models.CharField(max_length=120, null=True, blank=True)
            last_name = models.CharField(max_length=120, null=True, blank=True)
            email = models.EmailField()
            timestamp = models.DateTimeField(auto_now_add=True, auto_now=False)
            updated = models.DateTimeField(auto_now_add=False, auto_now=True)

            def __unicode__(self):
                return self.email

Finally the views.py is where you define in response to requests that are routed to the server. In alignment with the urls.py file the name of the functions on the views.py will match what is called in the urls.py file. For example in the urls.py we called the /learnmore page as follows: 

 url(r'^learnmore/$', views.learnmore, name="learnmore")
Now the corresponding code in the views.py file be as follows to catch the function: 
 
        def learnmore(request):
            return render(request, "learnmore.html")

Essentially what this does is whenever the learnmore function is called, it will take the request, create an HttpResponse object and then pass the learnmore.html in an HttpResponse object back to the user.

Ajax is a very important tool used in modern webpages. It allows a webpage to perform functions and update the page without reloading. For the purposes of this website Ajax was used to submit a request to the server, receive the response and update the page accordingly. Here is where Ajax makes a request to the server behind the scenes: 

 
        _makeAjaxRequest: function (sort_name, current_dataset) {
            var url = "/get_sort_result/";
            csrf_token = this._getCsrfToken()
            self = this
            $.ajax({
                type: "POST",
                url: url,
                dataType: "json",
                traditional: true,
                data: {
                    dataset: current_dataset,
                    csrfmiddlewaretoken: csrf_token,
                    sort_name: sort_name
                },
                success: function (response_data) {
                    if (response_data.error != "timeout") {
                        self._setSortedResult(response_data)
                    }
                    else {
                        self._setTimeoutError(response_data)
                    }
                },
                error: function (response_data) {
                    self._setDatasetError(response_data)
                }
            });

The .$.ajax function sets a call to the server, receives the response in the success or error function and updates the pages accordingly using JQuery.

JSON over REST

JSON is used as a way of organizing data to be passed over the web. JSON is used to break down objects into strings that can be read back by a different kind of platform. The great thing about JSON is that it allows a program to pass data to another program regardless of what is running on the other side as long as it accepts JSON messages. In the case of this website JSON is being passed from a JavaScript front end to a Django web server backend.

Here is an example of what a JSON message might look like as it is transmitted: 

 
             {\"array\": [1139, 2574, 4414, 7095, 8266, 9278], \"time\": 0.004565375466110427}"

In this example the data that is being passed is a variable called "array" with value [1139, 2574, 4414, 7095, 8266, 9278] and a variable named "time" with the value 0.004565375466110427. From the JavaScript side we can parse this JSON data with the following line: 

  JSON.parse(data);

Where data is the response object passed back by the server. On the Django side we can construct JSON objects to pass back to the client. Here is an example: 

 
            data = json.dumps({
            "array": returnarr,
            "time": elapsed_time
            })
            return HttpResponse(json.dumps(data), content_type='application/json

What we are doing here is constructing a JSON object using the json.dumps function, assigning variable names and assigning values to them. Finally we can return an HttpResponse object back to the client with the json parsed data.

REST which stands for Representational State Transfer is a communications protocol that allows us to pass data over the web. If you are familiar with web development then you probably know REST as the common HTTP messages GET, POST, PUT and DELETE. In this application we set up Django to be able to handle these types of messages using the following import

 from rest_framework import serializers

Also we want to set up our Django server to install the package requirements for REST

 pip install djangorestframework

Finally we need to add the rest_framework to our installed apps in the settings.py file

 
          INSTALLED_APPS = (
          ...
          'rest_framework'
          ...
          )

This is a great feature over typical web hosts as it gives us a lot more reach into the functionality of server. Below is an image of the console once one logs into Amazon Web Services




The way that we use AWS is that we find an instance that matches our needs, select the size of the server, deploy it and then log into it using SSH. In this case the kind of server that was used was the "Django - High-level Python Web Framework provided by TurnKey Linux" server. Linux Debian

Linux comes in many different flavors but for the Django server that Amazon Web Services provides it is a Debian distribution. Running our Django server on top of Linux instead of Windows is slightly different in how we want to set it up. Both Windows and Linux use the pip install django command in order to install Django so there is similarity there but installing package dependencies is different(and typically easier) in Linux.
